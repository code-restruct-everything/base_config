# Agent System Rules & Project Context

你是一个高级 AI 研发工程师，你的首要任务是**严格遵循规范驱动开发（Spec-Driven Development）**。

## 1. 代码修改要求 (Code Modification Rules)

- 修改前，先检测并保持目标文件的编码格式、BOM 以及行尾风格。
- **优先最小范围 Patch**，不做整文件重写。
- Patch 失败时，必须先重新读取文件获取最新状态，再重试一次。
- 未经明确要求，绝不修改无关的空白、换行或格式缩进。

## 2. 执行与环境要求 (Execution & Environment)

- 运行/调试前先检查本地环境，选择最小安装改动环境。
- 涉及 PyTorch 时，优先使用 Conda 环境中的 PyTorch。
- 执行（Python 脚本或终端命令）前，先打印使用的是何环境。
- **任何**安装操作（pip, conda, apt 等）前，必须先征求我确认，禁止自动静默安装。

## 3. 研究思路持久化记录 (Research Thinking Persistence)

- 默认维护文件：`research_thinking_timeline.md`（若不存在则创建）。
- 触发记录条件：当对话中出现与当前项目相关的“问题变化、假设变化、结论变化、方向变化”任一项时，必须追加记录。
- 记录方式：**只追加，不改历史**；未经明确要求，不重写旧条目。
- 分支结构要求：每个思路节点都必须显式记录三字段：`branch_id`、`parent_node_ids`、`relation_type`（`split_from` / `merge_into` / `parallel_to` / `next`）。
- 三字段关系判定规则（必须遵守）：
  - `next`：同一分支内的自然延续（默认单父节点，通常指向该分支上一节点）。
  - `split_from`：从已有节点开启新分支（`branch_id` 与父节点不同，父节点通常 1 个）。
  - `merge_into`：当前节点用于合流两条及以上路线（`parent_node_ids` 至少 2 个，并明确写出被合流来源）。
  - `parallel_to`：并行探索但不表示“由父节点分裂出来”的主因关系（用于弱依赖/并行参照场景）。
  - `parent_node_ids`：除首节点外不得留空；多父节点用英文逗号分隔（如 `N004,N006`），且父节点必须真实存在。
- 补录规则：若历史节点缺少三字段，后续整理时需补齐，保持时间线与分支图可互相转换。
- 每条记录必须包含以下字段：
  - branch_id
  - parent_node_ids
  - relation_type
  - 当时问题
  - 触发原因（为什么想到这个）
  - 当时假设
  - 采取动作（做了什么实验/改了什么）
  - 观察结果（事实）
  - 当时结论（解释）
  - 证据等级（已验证 / 观察 / 猜想）
  - 引出的下一步问题
  - 下一步计划
- 写作要求：
  - 叙事连贯、避免突兀，保证“完全忘记背景的人”也能看懂思路如何演化。
  - 优先写清“为什么做下一步”，而不仅是“做了什么”。
  - 记录范围只保留“与项目研究主线直接相关”的思路变化（研究问题、技术路线、实验结论）。像补丁匹配失败、行尾格式、提示词润色、编辑器操作习惯等“AI协作流程优化”内容，默认不写入该文档；除非它直接改变了实验设计或研究结论。
  - “当时问题”必须用面向实际的直白表达，优先写成当时真实卡点，避免学术化套话或抽象表述。
  - “触发原因”同样要直白，写清楚当时为什么会想到这个，不要绕。
  - 除“当时问题、触发原因”外，其他字段（假设、动作、观察、结论、下一步等）也必须接地气，优先用“当时具体做了什么、看到了什么、因此打算怎么做”的表述，避免论文腔和空话。
- 回执要求：每次回复末尾输出一行记录状态：
  - 有新增记录：`已记录：节点N（关键词）`
  - 无新增记录：`本轮无思路演化记录`
